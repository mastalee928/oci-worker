# 中转隧道模块 — 探讨记录（已搁置）

**状态**：搁置，后续再研究。**未授权实现**，开发仓勿擅自开工。  
**记录时间**：2026-05  
**触发**：用户曾考虑在 OCI Worker 增加类似极光/nyanpass 的中转模块；经讨论后决定暂缓。

---

## 用户意图演变

1. 最初：参考极光面板，做 **入口机 + 出口机 + 规则**，单用户，UI 简约，隧道加密做全；与 **Cloudflare、WebSSH 无关**。
2. 参考 nyanpass 官方文档（https://nyanpass.pages.dev/）后，了解其 **自研隧道 + nodeclient + gRPC 下发** 架构。
3. 核心动机其实是 **更好对付 GFW**；用户反馈 **极光方案下入口机经常被墙**。
4. 结论：**若主目标是过墙，在 oci-worker 里做「又一个转发面板」意义不大**；除非单独立项做抗探测入口，否则先不做。

---

## 已对齐的产品约束（若将来做）

| 项 | 结论 |
|----|------|
| 多用户/租用/倍率/商城 | **不做** |
| 与 CF / WebSSH | **完全独立模块** |
| 节点 | **入口机、出口机** 先登记，再 **自行添加规则** |
| 命名 | **用户自定义**，不预设「甲骨文入口」等地名/云商文案 |
| 加密 | 希望 **TCP/TLS/WS/WSS/KCP 等齐全**；实现路径待定（gost vs 自研） |
| 出口落地 | **单条 iptables** DNAT（出口机），与 gost 分工：加密在 gost/隧道层，iptables 只做 L4 落地 |
| 下发 | 讨论过 SSH 下发（极光式）与 **agent + 面板推送**（nyanpass 式）；未最终选定 |

## UI 线框共识（搁置前）

- 侧栏新模块，例：**「中转隧道」**；模块内 Tab：**规则 | 节点**。
- **节点页**：入口表 + 出口表；SSH、测试连接、入口可选端口池（如 10000–60000）。
- **规则页**：列表 +「添加规则」弹窗。
- **添加规则**：名称、入口、监听端口（留空随机）、出口（含「直接转发/不经出口」）、目标地址（多行格式待定）、**高级选项折叠**（对外协议、内链路 TLS 等）。
- 「刷新实例信息」式按钮对齐：规则区 **「刷新实例信息」** 与网络区 **「刷新网络信息」** 左对齐 —— *注：此为 InstanceList 已做功能，与中转模块无关。*

## 参考项目摘要

### 极光（Aurora-Admin-Panel/deploy）

- 多机 SSH 下发；gost / iptables / wstunnel 等多引擎。
- **借鉴**：节点 → 规则流程、协议选型思路。
- **不借鉴**：多用户租用、Docker 独立部署全家桶。

### nyanpass（商用闭源，文档公开）

- 架构：Go 面板 + **nodeclient** agent；规则 **gRPC 推送 + ~20s 同步**；失联不断流。
- 转发形态：**入口直出**（无隧道）、**双机隧道**、**用户单端**；双机专线可用 **iptables 仅接出口** 或 **双端接面板 + connect-host**。
- 协议：**自研 TCP 隧道**，外层 ws/http/tls；TLS 入站、SNI 分流、反向隧道、链式出口等（见 `/reference/connect/`）。
- 文档对 GFW **降低预期**：无协议能完全躲过 IP 筛查；ws 可能触发并发限制；CDN 穿墙已基本不可用。
- **单用户 oci-worker 可抄**：规则状态（未同步/正常/失败）、批量 JSON、出口 config JSON、一键对接命令；**不抄**：用户/倍率/商城。

## 自研隧道 vs gost（讨论结论）

- **自研协议栈**：能做 mux/特征控制，但 **难在长期对抗与运维**，不是「写完就过墙」。
- **入口被墙** 常见原因：裸 TCP、固定高端口、脏 IP、流量行为、境内入口 —— **换面板不能根治**。
- 务实路径（若将来做）：Phase1 **gost + iptables** → Phase2 可选自研 **入口↔出口** 内链路 → Phase3 抗探测（TLS 入站/SNI）；与 oci-worker 集成为 **可选 B**，非当前优先级。

---

## 搁置原因（用户原话归纳）

> 既然这样，做这个中转就没什么意思了，主要我是想对付墙好一点，用极光的话，入口机经常被墙。

助手结论：**过墙为第一优先级时，不建议在 oci-worker 开大中转模块**；精力可放在换入口策略（境外 443+TLS、IP、多入口）或现成抗探测方案，而非管理 UI。
